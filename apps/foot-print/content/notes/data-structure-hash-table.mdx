## 해시테이블 (Hash Table)

해시테이블은 **키(key)를 해시 함수로 변환해 배열의 인덱스로 쓰고, 해당 위치에 값을 저장**하는 자료구조다. 키-값 쌍을 저장하고 키로 빠르게 조회한다.

```text
key: "name"
         ↓ 해시 함수
index: 3
         ↓
버킷: [ _, _, _, "홍길동", _, _, _, _ ]
                  ↑
              index 3에 저장
```

### 해시 함수

해시 함수는 임의의 키를 고정된 범위의 정수(인덱스)로 변환한다. 좋은 해시 함수의 조건은 두 가지다.

- **결정론적**: 같은 키는 항상 같은 인덱스를 반환
- **균등 분포**: 인덱스가 특정 위치에 몰리지 않고 고르게 분산

### 시간 복잡도

| 연산 | 평균 | 최악 | 이유 |
|------|------|------|------|
| 삽입 | O(1) | O(n) | 해시 계산 후 바로 저장 / 충돌 심하면 탐색 필요 |
| 조회 | O(1) | O(n) | 인덱스로 바로 접근 / 충돌 심하면 탐색 필요 |
| 삭제 | O(1) | O(n) | 인덱스로 바로 접근 후 제거 / 충돌 심하면 탐색 필요 |

평균 O(1)이 가능한 이유는 해시 함수가 키를 인덱스로 바로 변환해주기 때문이다. 하지만 충돌이 많아지면 성능이 O(n)까지 떨어진다.

## 충돌 (Collision)

서로 다른 키가 같은 인덱스로 변환되는 현상이다.

```text
"name"  → 해시 함수 → index 3
"city"  → 해시 함수 → index 3  ← 충돌!
```

버킷의 수는 유한하기 때문에 충돌은 피할 수 없다. 어떻게 해결하느냐가 구현의 핵심이다.

### 체이닝 (Chaining)

같은 인덱스에 저장할 값이 있으면 **연결리스트로 이어 붙인다.**

```text
index 3: [ "name":"홍길동" ] → [ "city":"서울" ] → null
```

- 구현이 단순하고, 데이터가 많아도 유연하게 저장
- 충돌이 심해지면 연결리스트 탐색 비용(O(n))이 증가
- 포인터 저장 공간이 추가로 필요

### 개방 주소법 (Open Addressing)

충돌 시 다른 빈 슬롯을 찾아 저장한다. 모든 데이터가 배열 내에 저장된다.

**선형 탐사 (Linear Probing)**: 충돌 시 한 칸씩 이동해 빈 자리를 찾는다.

```text
index 3 충돌 → index 4 확인 → 비어있으면 저장
```

특정 구간에 데이터가 몰리는 **클러스터링** 문제가 생기기 쉽다.

**이차 탐사 (Quadratic Probing)**: 1², 2², 3²... 간격으로 탐색해 클러스터링을 완화한다.

**이중 해싱 (Double Hashing)**: 두 번째 해시 함수로 탐사 간격을 결정한다. 가장 균등하게 분산된다.

### 로드 팩터와 리해싱

**로드 팩터(Load Factor)** = 저장된 항목 수 / 버킷 수

로드 팩터가 높아질수록 충돌이 잦아져 성능이 떨어진다. 일정 임계값(보통 0.7~0.75)을 초과하면 **리해싱(Rehashing)**을 수행한다. 더 큰 배열을 새로 할당하고, 기존 항목을 전부 다시 삽입하는 작업이다.

```text
로드 팩터 0.75 초과
  → 버킷 크기 2배로 늘림
  → 기존 항목 전체 재삽입 (O(n))
  → 이후 평균 O(1) 회복
```

동적 배열의 확장과 같은 원리로, 분할상환 분석으로 보면 삽입은 여전히 평균 O(1)이다.

## JS에서의 해시테이블

### Object

JS 객체는 내부적으로 해시테이블로 구현돼 있다.

```javascript
const obj = {};
obj['name'] = '홍길동';  // O(1) 삽입
obj['name'];             // O(1) 조회
delete obj['name'];      // O(1) 삭제
```

**주의점:**
- 키는 **문자열 또는 심볼**만 가능 (다른 타입은 자동으로 문자열 변환)
- 프로토타입 체인으로 인해 의도치 않은 키가 존재할 수 있다 (`toString`, `hasOwnProperty` 등)
- 삽입 순서는 정수형 키를 제외하고 대체로 보장되지만 스펙에 명시된 건 아니다

```javascript
const obj = {};
obj[1] = 'a';
obj['b'] = 'b';
obj[2] = 'c';

Object.keys(obj);  // ['1', '2', 'b'] — 정수 키가 앞으로 정렬됨
```

### Map

키-값 쌍을 순서대로 저장하는 해시테이블이다. Object의 단점을 보완한다.

```javascript
const map = new Map();

map.set('name', '홍길동');
map.set(42, '숫자 키');
map.set({ id: 1 }, '객체 키');  // 객체도 키로 사용 가능

map.get('name');   // '홍길동'
map.has('name');   // true
map.delete('name');
map.size;          // 항목 수
```

**Object와의 차이:**

| | Object | Map |
|---|---|---|
| 키 타입 | 문자열·심볼만 | 모든 타입 |
| 삽입 순서 | 정수 키 예외 있음 | 항상 보장 |
| 크기 확인 | `Object.keys(o).length` | `map.size` |
| 이터러블 | 직접 불가 | `for...of` 가능 |
| 프로토타입 오염 | 있음 | 없음 |

**Map을 선택해야 할 때:**
- 키 타입이 문자열·심볼이 아닌 경우
- 삽입 순서가 중요한 경우
- 항목을 자주 추가·삭제하는 경우
- 키 개수를 자주 확인하는 경우

### Set

중복 없는 값의 집합이다. 내부적으로 해시테이블로 구현해 존재 여부를 O(1)에 확인한다.

```javascript
const set = new Set([1, 2, 2, 3, 3]);
set;          // Set {1, 2, 3} — 중복 제거
set.has(2);   // O(1)
set.add(4);
set.delete(1);
set.size;     // 3

// 배열 중복 제거
const unique = [...new Set([1, 2, 2, 3])];  // [1, 2, 3]
```

## 정리

- 해시테이블은 해시 함수로 키를 인덱스로 변환해 평균 O(1)에 삽입·조회·삭제
- 충돌은 피할 수 없다. 체이닝(연결리스트 연결) 또는 개방 주소법(다른 슬롯 탐색)으로 해결
- 로드 팩터가 높아지면 리해싱으로 배열을 확장해 성능을 회복
- JS에서 Object는 키가 문자열·심볼로 제한, Map은 모든 타입 가능하고 삽입 순서를 보장
- Set은 중복 제거와 O(1) 존재 여부 확인에 쓴다
