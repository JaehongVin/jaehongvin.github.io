## 배열 (Array)

배열은 **동일한 타입의 데이터를 연속된 메모리 공간에 순서대로 저장**하는 자료구조다.

핵심은 두 가지다.

- **연속된 메모리**: 요소들이 메모리상에 붙어 있다
- **인덱스 접근**: 시작 주소 + 인덱스로 위치를 바로 계산한다

### 메모리 구조

```text
인덱스:   [0]    [1]    [2]    [3]    [4]
값:       10     20     30     40     50
주소:    100    104    108    112    116  (int 기준 4바이트씩)
```

`i`번째 요소의 주소는 단순 공식으로 계산된다.

```text
address(i) = 시작주소 + (i × 요소크기)
```

이 계산이 상수 시간에 끝나기 때문에 인덱스 접근이 O(1)이다.

### 시간 복잡도

| 연산 | 복잡도 | 이유 |
|------|--------|------|
| 인덱스 접근 | O(1) | 주소 계산 한 번 |
| 탐색 (값 기준) | O(n) | 처음부터 순서대로 확인 |
| 끝에 삽입·삭제 | O(1) | 마지막 위치에 바로 저장·제거 |
| 중간·앞 삽입·삭제 | O(n) | 뒤 요소를 한 칸씩 밀거나 당겨야 함 |

중간 삽입이 O(n)인 이유는 명확하다.

```text
[10, 20, 30, 40, 50]  →  여기에 25를 삽입하려면

30, 40, 50을 한 칸씩 뒤로 밀어야 한다
[10, 20, __, 30, 40, 50]
[10, 20, 25, 30, 40, 50]
```

### 동적 배열

정적 배열은 선언 시 크기가 고정되어 있다. 현대 언어의 배열(Java `ArrayList`, Python `list`)은 크기가 가득 차면 **더 큰 배열을 새로 할당하고 복사**하는 동적 배열로 동작한다.

```text
capacity = 4  →  [10, 20, 30, 40]

5번째 요소 삽입 시도
  → capacity × 2 = 8짜리 새 배열 할당
  → 기존 요소 4개 복사  (O(n))
  → 새 요소 삽입

capacity = 8  →  [10, 20, 30, 40, 50, __, __, __]
```

복사가 발생하는 순간은 O(n)이지만, 용량을 2배씩 늘리는 전략 덕분에 **분할상환(amortized) 분석**으로 끝 삽입은 O(1)이 된다.

### JS 배열

JS 배열은 C처럼 타입이 고정된 정적 배열이 아니다. V8 엔진은 상황에 따라 두 가지 형태로 내부 표현을 바꾼다.

- **Fast Array**: 요소 타입이 동일하고 인덱스가 연속적이면 연속 메모리로 최적화
- **Dictionary Array**: 중간 인덱스가 비어 있거나(희소 배열) 타입이 섞이면 해시 테이블로 전환

```javascript
const fast = [1, 2, 3, 4];  // 연속 메모리 → Fast Array
const sparse = [];
sparse[9999] = 'z';          // 희소 배열 → Dictionary Array
```

주요 메서드 복잡도는 아래와 같다.

```javascript
arr.push(x)      // O(1) — 끝에 추가
arr.pop()        // O(1) — 끝에서 제거

arr.unshift(x)   // O(n) — 앞에 추가, 전체를 한 칸씩 밀어야 함
arr.shift()      // O(n) — 앞에서 제거, 전체를 한 칸씩 당겨야 함

arr.splice(i, n) // O(n) — 중간 삽입·삭제
arr.indexOf(x)   // O(n) — 선형 탐색
arr.includes(x)  // O(n) — 선형 탐색
```

## 연결리스트 (Linked List)

연결리스트는 **각 노드가 값(value)과 다음 노드의 주소(next)를 함께 저장**하는 자료구조다. 노드들이 메모리상에 연속으로 있을 필요가 없고, 포인터로 연결된다.

```text
[10 | •]──▶[20 | •]──▶[30 | •]──▶[40 | null]
  head
```

각 노드는 독립적인 메모리 공간을 가지며, `next`가 다음 노드를 가리킨다. 마지막 노드는 `null`을 가리킨다.

### 시간 복잡도

| 연산 | 복잡도 | 이유 |
|------|--------|------|
| 인덱스 접근 | O(n) | head부터 따라가야 함 |
| 탐색 (값 기준) | O(n) | head부터 순서대로 확인 |
| 삽입·삭제 (위치를 알 때) | O(1) | 포인터만 바꾸면 됨 |
| 삽입·삭제 (위치를 모를 때) | O(n) | 탐색 O(n) + 작업 O(1) |

삽입이 O(1)인 이유는 포인터만 교체하면 되기 때문이다.

```text
[10]──▶[20]──▶[30]  →  [20]과 [30] 사이에 [25] 삽입

[20].next = [25]
[25].next = [30]

[10]──▶[20]──▶[25]──▶[30]
```

배열처럼 뒤 요소를 밀 필요가 없다.

### 종류

**단일 연결리스트 (Singly Linked List)**

```text
[10 | •]──▶[20 | •]──▶[30 | null]
```

한 방향으로만 이동 가능. `next`만 저장한다.

**이중 연결리스트 (Doubly Linked List)**

```text
null◀─[• | 10 | •]⇄[• | 20 | •]⇄[• | 30 | •]──▶null
        head                          tail
```

`prev`와 `next` 두 포인터를 저장한다. 양방향 이동이 가능해서 삭제 시 앞 노드를 찾는 O(n) 탐색을 줄일 수 있다. LRU 캐시 구현에 주로 쓰인다.

**원형 연결리스트 (Circular Linked List)**

```text
[10 | •]──▶[20 | •]──▶[30 | •]──┐
   ▲                              │
   └──────────────────────────────┘
```

마지막 노드의 `next`가 `head`를 가리킨다.

### JS에서의 구현

JS에는 연결리스트 내장 자료구조가 없다. 객체로 직접 구현한다.

```javascript
const node = (value) => ({ value, next: null });

const head = node(10);
head.next = node(20);
head.next.next = node(30);
// 10 → 20 → 30
```

## 배열 vs 연결리스트

| | 배열 | 연결리스트 |
|---|---|---|
| 메모리 배치 | 연속 | 분산 |
| 인덱스 접근 | O(1) | O(n) |
| 중간 삽입·삭제 | O(n) | O(1) (위치 알 때) |
| 메모리 오버헤드 | 없음 | 포인터 저장 공간 |
| 캐시 효율 | 높음 | 낮음 |

배열은 **읽기(인덱스 접근)가 많을 때**, 연결리스트는 **중간 삽입·삭제가 잦을 때** 유리하다.

순회(traversal)는 둘 다 O(n)으로 같다. 오히려 배열이 캐시 지역성(cache locality) 덕에 실제 순회 속도는 더 빠르다.

삽입·삭제가 빈번하다고 무조건 연결리스트가 정답이 아닌 이유도 여기 있다. 캐시 미스 비용이 쌓이면 배열의 O(n) 이동보다 느려질 수 있다.

## 정리

- 배열은 인덱스 접근이 O(1)이지만 중간 삽입·삭제는 O(n)
- 연결리스트는 삽입·삭제가 O(1)이지만 인덱스 접근은 O(n)
- 순회는 둘 다 O(n)이고, 캐시 지역성 때문에 실제 속도는 배열이 더 빠른 경우가 많다
- JS 배열은 `push/pop`은 O(1), `shift/unshift`는 O(n)
- JS에서 연결리스트가 필요하면 직접 구현해야 한다
